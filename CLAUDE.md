# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

HALE (formerly RunLog; recursive acronym "HALE's Adaptive Life Engine", also "hale" as in hale and hearty): a self-hosted fitness tracker. FastAPI backend, SQLite storage (via SQLAlchemy), React+TypeScript frontend (`web/`, Vite + Tailwind + shadcn/ui — see "Frontend" below). Pulls activities from Strava (official OAuth API, primary) and optionally Garmin Connect (unofficial `garminconnect` library, secondary/manual-only), enriches them with historical weather, grade-adjusted pace, and a heuristic run-type/interval classifier, computed once at sync time and stored — not recomputed on read (except GAP, see Gotchas).

## Commands

- **Run it**: `docker compose up --build` from the repo root — app on `http://localhost:8000` (or `SYNC_INTERVAL_HOURS`/port as configured in `.env`, copy from `.env.example`). Same command whether running locally or on a remote host (NAS, VPS, etc.) — this project doesn't assume any particular deployment target. The `Dockerfile` is multi-stage: a `node:22-slim` stage builds `web/` into static assets, copied into the final `python:3.12-slim` image — Node is a build-time-only dependency, never present in the running container.
- **If you deploy to your own remote host**: keep host/path/SSH specifics in a local, gitignored file (e.g. `.RUNBOOK.md`) rather than committing them — those are personal to your setup, not project config.
- No test suite exists in this repo currently. The backend has no lint config; the frontend uses `oxlint` + `prettier` (run from `web/`, see below).

## Frontend (`web/`)

React 19 + TypeScript + Vite + Tailwind v4 + shadcn/ui, replacing an earlier vanilla-JS/Chart.js static frontend (`app/static/`) — that predecessor is kept reachable at `/legacy` for a short parity window after the cutover, then deleted; if `app/static/` is gone and this paragraph still is, that cleanup already happened and this note is stale. FastAPI serves the built `web/dist` at `/` via a catch-all route in `main.py` (`serve_web_app`) rather than a bare `StaticFiles(html=True)` mount, because that only auto-serves `index.html` at a mount's own root — a hard reload on a client-routed path like `/insights` needs an explicit fallback to `index.html` so React Router can take over. `/assets` (Vite's content-hashed JS/CSS bundle) is a plain `StaticFiles` mount.

API contracts are unchanged by the rewrite — same endpoints, same JSON shapes — so backend changes described below apply identically regardless of which frontend is live. Dev workflow (this repo's working copy is an SMB network share — see the gitignored `.RUNBOOK.md`): run `npm`/`vite` inside a throwaway `node:22-slim` container against the NAS-local path rather than over the mount directly, since native `fs.watch()` isn't supported over network filesystems and bulk `node_modules` operations are extremely slow over SMB.

## Architecture

**Two independent sync sources write to one `runs` table.** `app/strava.py` and `app/garmin_sync.py` each own their own fetch → normalize → upsert pipeline, keyed by a source-prefixed id (`"strava_<id>"` / `"garmin_<id>"`). They are **not** deduplicated in storage — both sources' raw records are kept forever, untouched, for data ownership. If both sources sync the same physical run, `web/src/lib/runs.ts`'s `mergeDuplicateRuns()` (ported 1:1 from the legacy frontend's identically-named function, called once per query in `hooks/useRuns.ts` before anything else consumes the result) merges them into one card at display time: matched by date/start-time/distance/activity-type proximity, preferring Strava's fields (richer route data) and falling through to Garmin's for anything Strava lacks (e.g. running-dynamics fields, which Strava never populates) — every downstream consumer (stat strip, Activities list, Insights, Map, activity filter) sees the merged result automatically. Strava is the auto-scheduled primary source (`app/main.py`'s `_auto_sync`, every `SYNC_INTERVAL_HOURS`, fires immediately on container start via `next_run_time=datetime.now()`); Garmin only runs when `POST /api/sync/garmin` is called manually.

**Derived metrics are computed at sync time, not read time**, and stored on the `Run` row: per-mile splits (resampled from raw GPS/HR/cadence streams for Strava, from lap data for Garmin), Grade-Adjusted Pace (Minetti cost-of-running model), heat index / wet bulb (NWS Rothfusz regression / Stull approximation, from Open-Meteo's hourly humidity), run-type classification and interval/rep detection (`app/util.py`, pace-variability heuristics) — all in `app/strava.py`/`app/garmin_sync.py` at sync time. **Exception:** GAP is *also* independently reimplemented client-side in `web/src/lib/gap.ts` (`gapSecPerMi`/`minettiCost`) to redraw per-split GAP without a round trip — all Minetti implementations (backend `app/util.py`, frontend `web/src/lib/gap.ts`) must be kept in sync by hand if the formula ever changes.

**No migration framework.** `app/models.py`'s `init_db()` calls SQLAlchemy's `create_all()` (which only creates missing *tables*, not missing *columns*) followed by `_migrate_add_missing_columns()`, a hand-rolled `PRAGMA table_info` diff + `ALTER TABLE ADD COLUMN`. Any new column added to the `Run` model needs no separate migration step — this function picks it up automatically on next container start — but the type-mapping in that function (TEXT/INTEGER/REAL) needs to stay correct for new column types.

**`sync_meta` table** persists last-sync timestamp/count/error per source (`get_sync_meta`/`set_sync_meta` in `models.py`), written by `main.py`'s `_record_sync()` after every sync attempt (auto or manual), read via `GET /api/sync/meta`. This is what backs the "Last synced" UI text — it intentionally does not reset on page reload or container restart.

**Garmin is explicitly best-effort.** There's no official Garmin API; `garminconnect` logs in with real credentials and can break without warning. Pinned to `0.3.6` (upgraded from `0.3.2` — that upgrade is what forced the Dockerfile to `python:3.12-slim`, since `0.3.3+` requires Python 3.12; `0.2.19` before that pulled in a broken transitive `withings-sync` dependency chain, avoid going back below `0.3.x`). `garmin_sync.py`'s login caching (`_login`/`_lightweight_session_check`/`_harden_tokenstore_permissions`) reaches into `garminconnect`'s private internal client (`client.client` — `is_authenticated`, `_token_expires_soon()`, `_refresh_session()`, `dump()`/`load()`, `di_refresh_token`) to avoid calls the public `login()` wrapper makes unconditionally; these are underscore-prefixed/undocumented, so re-verify they still exist before bumping this pin again. 429s from Garmin during login surface as a raw `TypeError` from the client library rather than a clean rate-limit error; `main.py`'s `manual_sync_garmin` pattern-matches the message to give a readable error instead.

**Strava OAuth** is a live in-app flow (`/auth/strava/login` → Strava → `/auth/strava/callback`), not a manual token exchange — tokens are stored in the `oauth_tokens` SQLite table and auto-refreshed in `strava.get_valid_access_token()`.

**AI chat assistant (Chat tab)** is grounded in real data, not a roleplay prompt. `app/stats.py` is the single computation core for both the dashboard endpoint and the chat tools — never add a second implementation of a stat; extend `stats.py` and have both call it (same discipline as GAP above, which *is* duplicated and is a cautionary example, not a pattern to repeat). `app/assistant.py` wraps each `stats.py` function as a read-only tool via the Claude Agent SDK's `create_sdk_mcp_server`/`@tool`, with `allowed_tools` + `disallowed_tools` + `setting_sources=[]` as layered defense against the agent touching anything beyond its own tools (it runs unattended, server-side, in the same container as the DB). Auth is either `CLAUDE_CODE_OAUTH_TOKEN` (Claude Pro/Max subscription, no metered billing) or `ANTHROPIC_API_KEY` (standard pay-as-you-go) — both optional, checked at runtime via `assistant.is_configured()`; neither set just disables the Chat tab's input cleanly. The `claude-agent-sdk` pip package bundles its own CLI binary per-platform (confirmed for `manylinux_2_17_x86_64`, matching this app's `python:3.12-slim` container) — this runs entirely at request time in the Python container and needs no Node.js itself; Node only exists in the separate frontend build stage (see "Frontend" above), never in the final runtime image.

**Multi-user data foundation exists, but there's no login yet.** `User`/`ProviderCredential` tables (`app/models.py`) hold per-user Strava/Garmin credentials — `strava.py`/`garmin_sync.py` take a `user_id` everywhere, and `Run`/`DailySteps`/`ChatMessage`/`Goal` all carry a `user_id` column. In practice every request today uses the hardcoded constant `DEFAULT_USER_ID` ("default") — there's no session/auth layer yet, so treat any endpoint as single-user until real login ships. When filtering by `user_id`, always use `models.owned_by(Column, user_id)` rather than `Column == user_id` directly: rows synced before this migration have `user_id=NULL`, and `owned_by()` is what correctly treats those as belonging to the default user (matching this repo's established pattern of handling legacy-NULL at read time, e.g. `r.recovery_json or "[]"`) without also matching NULLs for any other real user. `DailySteps` still has a plain `date` primary key (not composite with `user_id`) — a real limitation, not yet fixed, see STATUS.md.

**Goals** (`app/models.py`'s `Goal`, one wide table for all three types — race/consistency/distance_target) drive both the Home tab's goal cards and the header's race countdown (`web/src/components/layout/RaceCountdown.tsx`, which reads `daysUntil` from `/api/goals`' already-computed `progress` field rather than recomputing date math client-side — pulls the nearest active race-type goal). `stats.goal_progress()` deliberately never outputs an invented "on track" judgment — only real countdown/volume/percentage numbers — same "don't fabricate a score" principle as the dashboard cards.
