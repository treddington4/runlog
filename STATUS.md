13. **Bug fix: location labels re-geocoded slowly on every refresh** (this session, same day) — root cause was `geocodeClustersInBackground()` in `app.js` sleeping ~1.1s after *every* cluster regardless of whether it was a `localStorage` cache hit or an actual Nominatim call — so even a fully-cached refresh crawled through labels one at a time. Fixed by moving geocoding server-side: new `GET /api/geocode?lat=&lon=` in `main.py` caches by rounding to 2 decimals (`f"geocode_{lat:.2f}_{lon:.2f}"`) and reusing `sync_meta`'s existing key/value get/set helpers (no new table/migration needed) — the cache is now DB-backed and shared across every browser/device instead of per-browser, and a cache hit is a single fast same-origin round trip with no artificial delay at all. 14. **Reworked Pace/HR/Cadence heatmaps: color-shifting route lines instead of blob heatmap** (this session, same day, user feedback: "the current one doesn't look right") — `leaflet.heat` is built for point-density heatmaps and renders diffuse blurred blobs, which loses the actual route shape entirely; not what "color shift to show things" meant. Replaced with `app.js`'s `buildMetricSegments()`: every consecutive pair of `routeMetrics` points becomes a 2-point line segment carrying that segment's metric value, bucketed into 16 color bins (new `heatColor(t)` — a blue→cyan→green→yellow→red gradient) and drawn as 16 `L.polyline` layers (one per bucket, an array-of-segments each) inside an `L.featureGroup` — a handful of Leaflet layers regardless of point count (was a concern at ~43k segments across full history), and `getBounds()` works again for fitting since `FeatureGroup` has it (unlike `HeatLayer`). Removed the now-unused `leaflet.heat` CDN script. **Also fixed a real color-scale bug found while sanity-checking this against real data before deploying**: bucketing on raw min/max let a handful of near-stopped/GPS-noise points (`paceSecPerMi` up to 5328, i.e. ~88 min/mi, right at the `speed > 0.3` m/s cutoff) stretch the scale so far that normal running-pace variation (mostly 400–1000 sec/mi) got crushed into 1–2 of the 16 buckets — nearly monochrome in practice, likely the actual substance of "doesn't look right" even before the blob-vs-line issue. Switched to clipping at the 5th/95th percentile (values outside that range clamp into the extreme bucket rather than stretching the scale). Verified via a Node simulation against the real 43,444 pace segments: raw min/max gave a range of 278–5329 sec/mi with 92% of segments packed into 2 buckets; percentile clipping gives a 497–988 sec/mi (8:17–16:28/mi) range spread reasonably across all 16 buckets. The on-map legend now also shows the actual value range (e.g. "blue 9:40/mi → red 6:15/mi") using the existing `paceStr()` formatter plus new bpm/spm formatters, instead of a generic "blue = slower" caption.

## Known gaps / things to watch

- **No test suite** exists in this repo.
- **`DailySteps` still has a plain `date` primary key, not composite `(date, user_id)`** (item 39) — safe today since there's only one real user, but two real users syncing steps for the same calendar date would overwrite each other. Needs a real migration (table recreation, not a simple `ALTER TABLE`) before true multi-user step tracking works.
- **`sync_meta` (including `garmin_resting_hr_bpm`, the HR-plausibility floor's source) is still a single global table, not per-user** (item 39, deliberately out of scope) — fine with one real user, would need scoping before a second real user's resting HR/last-sync info stops being shared.
- **No login, no password enforcement, no encryption at rest** (item 39, deliberately deferred by explicit user request — "I don't mind if login and passwords come later") — the multi-user *data* foundation is real and verified, but anyone with network access to the app today acts as the one `default` user. `User.password_hash` exists but is unused. Don't expose this app beyond a trusted LAN until real auth ships.
- **GAP (grade-adjusted pace) is implemented twice** — once in `app/util.py` (backend, used at sync time) and once in `app/static/app.js` (frontend, used for live split recalculation). If the Minetti formula ever changes, both need updating.
- The app is plain HTTP on `:8000`, no TLS/reverse proxy — fine on LAN, would need attention before any exposure beyond the local network.
- I haven't been able to visually verify the UI myself (the sandboxed browser can't reach the NAS's private LAN address — confirmed again this session, see `.RUNBOOK.md`) — verification so far has been via `curl`/API checks plus Node-based logic simulation for the date-filter math. Worth a visual once-over, especially the new Settings tab, custom date picker, and the new Backlog Sync log panel.
- **Backlog sync job state is in-memory only** — if the container restarts mid-run, the live progress/log is lost (though any activities already committed stay committed, since it commits per-activity, not just at the end). The "last completed" summary in Settings persists fine via `sync_meta`. Not worth a DB-backed job table for a single-user LAN app; noting it in case it's surprising later.
- Strava backlog sync makes ~2 API requests per activity (streams + sometimes laps) plus one per page — for very large histories this could approach Strava's rate limits (100/15min, 1000/day for default apps). Not hit in practice yet (327 activities synced fine), but worth knowing if a future account has thousands of activities.
- ~~Garmin route capture is unverified against the live API~~ — **verified by item 31**: real Garmin runs in the DB have real `geoPolylineDTO`-derived routes (269 points on the inspected example).
- A residual few Strava activities have no route data by *either* path (`route`/`summary_polyline` or `routeMetrics`/streams) — likely activities with no GPS at all on Strava's side (e.g. very old/manually-entered/indoor-but-not-flagged-treadmill entries), not a bug, not counted precisely after the routeMetrics fallback (item 19) changed the denominator.
- Map tab pulls tiles from a public CartoDB CDN with no API key — fine for personal/low-traffic use, but there's no fallback if that free tier ever throttles.
- Location clustering is a simple greedy proximity pass (order-dependent, first-cluster-within-radius-wins) — fine at ~200 runs/handful of locations, would need a real clustering algorithm (k-means, DBSCAN) if the location count ever got large. Reverse-geocode labels now cache server-side in `sync_meta` (see item 13), so every browser/device benefits from the same cache.
- The `sync_meta` table is being used as a general small key/value cache now (sync bookkeeping *and* geocode labels) — fine at this scale, but if it ever grows large a dedicated cache table/TTL would be cleaner. Geocode cache entries never expire — correct for place names (they don't change), but means a bad/wrong label would need a manual DB fix (`DELETE FROM sync_meta WHERE key LIKE 'geocode_%'` to force re-lookup), not a UI action.
- ~~Pace/HR/Cadence heatmaps are Strava-only~~ — **addressed by item 41**, Garmin runs now get `routeMetrics` populated from the raw FIT file's `record` stream, same shape as Strava's. Still needs a real sync to confirm it actually works end-to-end (see item 41's unverified checklist) before calling this fully resolved.
- **Garmin FIT `record`-message field names are unverified against a real live file** (item 41) — the parsing code follows the public FIT SDK profile and degrades gracefully per-field, but hasn't been exercised against this account's actual hardware yet (Garmin's been rate-limited all session). First real sync should be checked against the item 41 checklist (plausible lat/lon range, `route_source` actually shows `"fit_record_stream"`, the known duplicate pair's FIT-derived start/end lands further from Strava's masked point, not closer).
- Routes with fewer points don't line up 1:1 between `route` (from `summary_polyline`) and `routeMetrics` (from the `streams` endpoint) — a few runs have one but not the other. As of item 19, density mode also falls back to deriving a route from `routeMetrics` when `route` is empty, so this mismatch matters less than it used to, but the two fields are still independently sourced and can diverge.
- `HEAT_BUCKETS = 16` and the `heatColor()`/`GRADIENTS` stops (items 14/15/17) were tuned against Node simulations of the real distribution, not visually in a browser — the bucket *counts* are verified sane for all four metrics now (pace, HR, cadence, grade), but nobody has actually looked at the rendered colors yet. Worth a visual check.
- **Insights/stat-strip's "Run-only" filtering (item 18) is a simple `activityType === "Run"` check** — doesn't account for e.g. a trail run that Strava tagged differently, or future activity types that should arguably count as running (treadmill walks that are really run/walk intervals, etc). Fine for now given the explicit "don't build anything fancy yet" scope, but worth revisiting if the type list grows.
- **Daily steps and resting HR still have never been seen with real data**, even though route capture and running dynamics now have (item 31) — `/api/steps` still returns `[]` and `/api/config`'s `restingHrBpm` is still `null`. Either those two sync calls didn't fire during whatever window Garmin was briefly reachable, or they have their own separate bugs not yet exposed. Worth a specific check (not just "did Garmin sync at all") the next time a sync succeeds.
- HR and cadence heatmaps still use symmetric 5th/95th percentile clipping on both ends (item 15 only changed pace) — untested assumption that they don't have the same "real extreme gets erased" problem pace had. If someone has a genuinely notable HR/cadence spike that looks washed out the same way the sprints did, the fix is the same one-line change: set `clipMin`/`clipMax` to `false` on whichever end matters for that metric in `METRIC_CONFIG`.
- ~~Running dynamics is the least-verified thing added this session~~ — **the core FIT-parsing path is now verified by item 31** (real, plausible values on a real run). Field-name coverage beyond the 5 fields already seen is still unconfirmed (e.g. if this account's hardware records other running-dynamics fields `_fetch_running_dynamics()` doesn't currently ask for). Still the heaviest addition to Garmin's per-activity API load (main activity + laps + geoPolyline details + a full FIT file download, per run) — worth watching whether this contributes to rate-limit pressure going forward, now that item 30's token caching should otherwise reduce it.
- **Recovery time (item 29) is only ever computable for runs with genuine lap-level work/recovery structure, which is most of them, currently just 1 of 17 "Interval"-classified runs.** `classify_run_type`'s Interval label (mile-split pace variance) and `detect_intervals`'s work/recovery labels (lap duration + pace vs. median) are independent heuristics — a run can be "Interval" overall without its Strava laps actually being lapped per rep (auto-lap by mile/time is common and doesn't produce clean work/recovery boundaries). Not a bug in this feature, but means recovery data will look sparse across most of this account's history unless the athlete starts manually lapping interval reps going forward, or `detect_intervals` itself gets revisited to work from mile splits instead of laps when laps aren't rep-aligned.
- **Garmin token persistence (item 30) hasn't actually been confirmed to work end-to-end** — the save/load mechanics were verified directly against the installed `garminconnect`/`garth` library source, and the fallback path (real login when no cached session exists) was exercised once and behaved correctly (still 429'd, as expected given the pre-existing cooldown), but the *success* path — a real login succeeding, tokens being saved, and a *subsequent* sync loading them and skipping the login endpoint — has never actually happened yet, since every attempt this session has 429'd. First real Garmin sync should confirm `/data/.garmin_tokens` gets created with both token files, and that a second sync shortly after doesn't trigger a new login in the container logs.
- **Duplicate-run merging (item 32) matches on date + time + distance + type proximity, not a hard identifier** — could theoretically produce a false-positive merge if two genuinely different short runs happened on the same day within 10 minutes of each other at nearly the same distance (unlikely, but not impossible for someone doing e.g. two similar-length loop repeats). All 3 real matches found this session were manually reviewed and are unambiguous (exact distance to 3 decimal places), so this hasn't been an issue in practice. The merge also only ever considers pairs *within a single API response* — if Strava and Garmin sync at very different times such that one of the pair isn't in `runs` yet when the other loads, they just won't be merged that particular page load (not a bug, just a limit of doing this client-side per-request rather than server-side/persistent).
