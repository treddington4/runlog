# Status

_Written for picking this project back up in a fresh session — see [CLAUDE.md](CLAUDE.md) for architecture._

## Current state

- **Deployed and running** via standard `docker compose up --build` (see [README.md](README.md) for setup). Any Docker host works the same way — nothing in this project assumes a particular deployment target. If you deploy to your own remote host, keep host/path/SSH specifics in a local, gitignored file (e.g. `.RUNBOOK.md`) rather than committing them.
- **Git**: [github.com/treddington4/runlog](https://github.com/treddington4/runlog), `master` branch.
- **Strava**: connected and auto-syncing (official OAuth, every `SYNC_INTERVAL_HOURS` + immediately on container start). Captures every activity type, not just runs (Run, Ride, Walk, WeightTraining, Yoga, Workout, Swim, etc.) — running-specific classification only applies to `Run`-typed activities.
- **Garmin**: optional, manual-only sync via the unofficial `garminconnect` library (see the "Garmin is explicitly best-effort" note in CLAUDE.md). Session tokens are cached to disk after login so a normal sync doesn't repeat a full credential login every time — only a real rate-limit block forces a fresh one.

## Architecture notes and gotchas

These are the non-obvious decisions worth knowing before touching related code. See `CLAUDE.md` for the higher-level architecture; this is the "why it's built this way" layer.

- **Raw data is never mutated to "fix" bad-looking values — only display/aggregation filters it out.** A dead HR sensor, a GPS/distance glitch, or a corrupted single reading all stay in the DB exactly as the source reported them (data-ownership principle). `app/stats.py`/`app/static/app.js` both implement matching plausibility filters (`isPlausibleHR`, `isPlausiblePace`) applied at every aggregate/chart/display site, not at ingestion.
- **HR floor is dynamic, not hardcoded**: if Garmin has synced a resting HR, the "implausible HR" floor is `restingHR - 10%`; otherwise it falls back to a Strava-history-derived proxy (5th percentile of valid readings, minus 10%). Example: a resting HR around 45bpm implies anything under ~40bpm during exercise gets treated as a sensor glitch, not real physiology.
- **Duplicate Strava+Garmin activities are merged only at display time**, never in storage — both sources' rows are kept forever. `mergeDuplicateRuns()` (`app.js`, called once in `loadRuns()` before anything else consumes `runs`) matches by date/start-time/distance/type proximity, prefers Strava's fields (richer route/heatmap data), falls through to Garmin's for anything Strava lacks (e.g. running-dynamics fields, which Strava never populates), and prefers Garmin's activity *name* specifically (tends to be more descriptive than Strava's auto-generated names). A small badge marks merged cards so this is visible, not silently hidden.
- **Map/heatmap system**: `Run.route_json` stores decoded GPS points (free from Strava's `summary_polyline`; best-effort from Garmin's `geoPolylineDTO`). `Run.route_metrics_json` stores a decimated (~300 point) geo-tagged series of pace/HR/cadence/grade, built from the same streams call already used for splits — no extra API cost. Heatmap color bucketing clips outliers at the 5th/95th percentile per metric, but **asymmetrically for pace**: only the slow end clips (near-stopped/GPS-noise artifacts), because the fast end is real, meaningful effort (genuine sprint splits) that symmetric clipping would otherwise wash into the same bucket as ordinary tempo pace. Routes are split into separate polyline segments wherever a GPS gap exceeds an adaptive per-route threshold (median point-spacing × 6), so a paused-and-relocated run doesn't draw a straight line across the gap. The Map tab defaults to the location cluster of the most recent activity (not "all locations" zoomed out) — deliberately *not* browser geolocation, since `navigator.geolocation` requires a secure context (HTTPS/`localhost`) that a plain-HTTP LAN deployment doesn't satisfy.
- **Running dynamics** (vertical oscillation, ground contact time, stride length, power) come from parsing the athlete's raw Garmin `.FIT` file directly (`fitparse`), not any summary API — one extra per-activity API/download call, only made for `Run`-typed Garmin activities given Garmin's rate-limit sensitivity.
- **Interval recovery time** is defined as: seconds from a work rep's peak HR until HR drops 20bpm within the following recovery rep (a relative drop, not a fixed target HR). Only computable for runs whose Strava laps actually delineate real work/recovery reps — most auto-lapped-by-mile runs don't, so this stays sparse across most histories unless reps are manually lapped.
- **Sleep timestamps from Garmin have no timezone marker in the raw JSON** (e.g. `"2026-07-13T03:26:54.0"`) even though they're genuinely UTC — per the ECMAScript spec, a timezone-less date string parses as *local browser time*, silently shifting the whole night by the browser's UTC offset. `parseUtcTimestamp()` in `app.js` appends `Z` before parsing unless the string already carries a timezone marker.
- **Race goals auto-link to a matching Run** once the target date passes (`Goal.linked_run_id`, `stats._find_and_link_race_run()`): searches for a Run within ±1 day of the goal's date among its configured activity types, prefers the closest distance match. Leaves the goal "active" (not "completed") if nothing matches — never fabricates a success verdict. Same "don't invent a score" discipline applies to `goal_progress()` generally: it only ever reports real countdown/volume/percentage numbers, never an "on track" judgment.
- **AI chat assistant** (`app/assistant.py`) is grounded in real data via read-only tools wrapping `app/stats.py` — never a second implementation of a stat (same discipline as GAP, which *is* duplicated between backend/frontend and is the one accepted exception). Defense-in-depth: `allowed_tools` allowlists only the stats tools, `disallowed_tools` blocklists the SDK's built-in filesystem/shell tools, `setting_sources=[]` prevents any local config from granting anything back — it runs unattended, server-side, in the same container as the DB.
- **Multi-user data foundation exists without login yet.** `User`/`ProviderCredential` hold per-user credentials; `Run`/`DailyWellness`/`ChatMessage`/`Goal` all carry `user_id`. Rows from before this migration have `user_id=NULL` — always filter with `models.owned_by(Column, user_id)`, which treats `NULL` as belonging to the default user without also matching `NULL` for any other real user.
- **Garmin export ZIP import** (`app/garmin_import.py`, for backfilling history without hitting the live rate-limited API) required correcting several undocumented unit/format quirks found by inspecting a real export: `distance`/`elevationGain` are in centimeters (100x a naive meters read), `duration` fields are in milliseconds, timestamps are epoch-millisecond numbers (not ISO strings), and the exported raw `.FIT` files can't be matched back to activity IDs by filename at all (that matching path was removed rather than left in as dead code).

## Known gaps / things to watch

- **Activity-type matching for stats/goals is an exact string match** — a `["Run"]`-typed goal or stat silently excludes Garmin-sourced runs, since Garmin's raw type is lowercase (`"running"`) vs. Strava's `"Run"`. Only normalized narrowly inside the race-goal-linking helper so far; a broader fix would change every existing goal/dashboard number and needs its own verification pass.
- **Activities imported via the export-ZIP path have no route/GPS/splits/running-dynamics data** — the raw `.FIT` files in the export can't be matched to activity IDs, and the import path doesn't call the live laps/route API. Backfilling that data requires a deliberate live re-sync (already-imported activities are skipped by normal dedup).
- **No test suite** exists in this repo.
- **`DailyWellness` (daily steps/resting HR/etc.) has a plain `date` primary key, not composite with `user_id`**: fine with one real user, but two users syncing the same calendar date would overwrite each other. Needs a real migration before true multi-user wellness tracking works. `sync_meta` (used for cross-sync bookkeeping, including the Garmin resting-HR floor) is similarly a single global table, not yet per-user.
- **No login, no password enforcement, no encryption at rest** — deliberately deferred. `User.password_hash` exists but is unused. Anyone with network access to the app acts as the single default user; don't expose this beyond a trusted network until real auth ships.
- **GAP (grade-adjusted pace) is implemented twice** — once in `app/util.py` (backend, sync time) and once in `app/static/app.js` (frontend, live split recalculation). If the Minetti formula ever changes, both need updating.
- Plain HTTP, no TLS/reverse proxy built in — fine on a trusted LAN, needs attention before any exposure beyond that.
- **Backlog sync job state is in-memory only** — a container restart mid-run loses the live progress/log (though already-committed activities stay committed, since it commits per-activity). The "last completed" summary persists fine via `sync_meta`.
- Strava backlog sync makes roughly 2 API requests per activity — for very large histories this could approach Strava's rate limits (100/15min, 1000/day for default apps). Not an issue at moderate history sizes.
- Map tab pulls tiles from a public CartoDB CDN with no API key — fine for personal/low-traffic use, no fallback if that free tier ever throttles.
- Location clustering is a simple greedy proximity pass (order-dependent) — fine at a few hundred runs across a handful of locations, would need a real clustering algorithm (k-means, DBSCAN) at larger scale. Reverse-geocode labels cache server-side in `sync_meta` so every device benefits from the same cache.
- Recovery-time data (see above) will look sparse across most histories unless interval reps are manually lapped, or `detect_intervals` gets reworked to derive reps from mile splits instead of laps.
- Duplicate-run merging matches on date + time + distance + type proximity, not a hard identifier — a false-positive merge is theoretically possible for two very similar short runs on the same day within ~10 minutes of each other, though not observed in practice.

## Possible next steps (not started, just noted)

- Visual QA pass across the UI — most of this project's verification has historically been via API/curl checks and logic simulation rather than a real browser session.
- Real weight/strength-training tracking (exercise/set/rep/weight detail, a dedicated view, etc.) — not scoped yet, needs its own design conversation before building. Garmin's live API likely has the underlying set/rep/weight data; unclear whether the export ZIP also carries it.
- If Garmin sync activity is reliable going forward, bring pace/HR/cadence/grade heatmaps to Garmin-sourced runs at parity with Strava's.
- Broader activity-type support beyond "capture and list" (dedicated cycling/hiking metrics, a real classification scheme) — deliberately out of scope for now.
- Database backup strategy for the persistent volume beyond the default Docker-managed volume.

See also the "Sync engine rework" plan (dedup via a `detail_synced_at` marker, expanded Garmin wellness metrics — HRV, VO2max, sleep score, stress, SpO2, hydration, weight — and a unified per-source backoff sync coordinator) for the next planned body of work.
